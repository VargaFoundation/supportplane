1. About this document

There is absolutely no question that Hadoop is a fundamentally disruptive technology. New advancements — in scalability, performance, and data processing capabilities — have been hitting us every few months over the last 4 years. But as Hadoop becomes the new normal IT teams, developers, and security practitioners are playing catch-up to understand Hadoop security and use efficently the computational resources available. This document lays out a series of recommended security and performance controls for Hadoop.

1.1. Context

This report is based on the Hadoop cluster of the EY Lab team. The actual cluster is composed of 6 hosts:

  • 1 edge node

  • 1 master node

  • 4 data node

1.2. Scope of the report

The report covered the following:

  • Hardware, network, security design survey and analysis

  • Review current and future Hadoop requirements

  • Review the Hadoop architecture, ingestion pipeline, data schema, and data partitioning system

  • Review any data processing jobs or analytics processes

  • Review the data-serving and results-publishing systems

  • Run standard and possibly specialized performance tests

  • Recommend performance tuning, data compression and decompression, and scheduler configuration

  • Review management and monitoring processes and production procedures

1.3. Approach and methodology

A set of tools and program are executed on the cluster in order to gather informations about the hosts and the components. These informations are then collected, processed and reviewed, either by automatic procedures or directly by an expert.

1.4. Explanations

Each control is identify by a flag, one or more threats, one or more vulnerabilities, one or more assets, potential impacts, a likelihood, a risk level and some recommendations.

A threat refers to a new or newly discovered incident with the potential to do harm to a system or your overall organization. A vulnerability refers to a known weakness of an asset (resource) that can be exploited. Recommandations refers to advisories, solutions, and tools. Risk translates to the following:

risk = likelihood x impact

Example:
Threat

  • Sensitive data leakage (Man in the middle)

Vulnerability

  • Lack of access control policy

Asset

  • Data

  • Servers

Impact

Critical data may be stolen or corrupted

Likelihood

Medium

Risk

High
Recommandations

  • The first action to do in order to fix this

  • An other possible action

Each control is marked with an icon, corresponding to the effective risk of the control on the actual state of the cluster:

The current state of the cluster is good for this control, or some minor corrective actions may be needed.

The current state of the cluster is not fully compliant for this control, a plan for corrective measures should be developed within a reasonable period of time.

The current state of the cluster is critical regarding this control, a plan for corrective measures should be developed as soon as possible.

During the audit process, we were unable to determine the actual state of the cluster on this control.

2.2. Naming, aliases and CNAMES

Use CNAMEs to identify your servers to the cluster so that you can rename and renumber your servers without downtime. A Canonical Name record (abbreviated as CNAME record) is a type of resource record in the Domain Name System (DNS) used to specify that a domain name is an alias for another domain, the "canonical" domain.

Example: data001,zk002,primary002,master001, instead of labcentos17.freylab.net

This setting avoid reconfiguring applications (hdfs/hbase/core-site.xml) when changes occur on the cluster.
Threat

  • Long service downtime

Vulnerability

  • Incapacity of restoring a single server or a service on a single server

Asset

  • Servers

Impact

Long cluster downtime

Likelihood

High (OS crashes, server reboot and other production issues will certainly happen)

Risk

Medium (batch-oriented cluster, not real-time platform using Kafka/Storm…​)
Recommandations

  1. Update DNS with logical name. Use one logical per service/role/component

2.4. HDFS

2.4.1. Small files

A small file is one which is significantly smaller than the HDFS block size (default 64MB). Reading through small files normally causes lots of seeks and lots of hopping from datanode to datanode to retrieve each small file, all of which is an inefficient data access pattern.
Threat

  • Failure of information system

Vulnerability

  • Inadequate supervision

Asset

  • HDFS

Impact

Current disk usage for HDFS is 24.3 GB, for a total capacity of 1.9 TB. Around 5000 blocks are used, of an average size of 5 MB, which is far less than the default block size. Although the actual size used on HDFS is not that big, leaving this behavior on the cluster may introduce performance issues in the future.

Likelihood

Low

Risk

Medium
Recommandations

  1. Use HDFS archive

  2. Don’t use partitionning when data are small

  3. Use HBase if you have many small data

2.5.2. Container isolation

CGroups is a mechanism for aggregating/partitioning sets of tasks, and all their future children, into hierarchical groups with specialized behaviour. From a YARN perspective, this allows containers to be limited in their resource usage.

Via Ambari, this can be done via the interface by clicking on "CPU Isolation"

3. Security

3.1. Data Protection & Privacy

3.1.1. At rest

Transparent data encryption

HDFS implements transparent, end-to-end encryption. Once configured, data read from and written to special HDFS directories is transparently encrypted and decrypted without requiring changes to user application code.
Threat

  • Information leakage

Vulnerability

  • Inadequate change management

  • Lack of access control policy

  • Inadequate control of physical access

Asset

  • HDFS

  • Data

Impact

Attacks at the filesystem-level and below (so-called “OS-level attacks”) are possible Disks often need to be removed from the cluster and replaced. Human readable residual data remains when data is removed or when disks are decommissioned.

Likelihood

Medium or High (depending of the underlying storage system of the virtual host)

Risk

High
Recommandations

  1. Install an configure Hadoop Key Management Server (KMS)

  2. Define encryption zones for sensitive data

3.1.2. Wire encryption

RPC Encryption

The data transfered between Hadoop services and clients can be encrypted on the wire.
Threat

  • Information leakage

Vulnerability

  • Unprotected network connections

  • Inadequate control of physical access

Asset

  • All (servers and clients)

Impact

Communications between Hadoop services and clients are not encrypted and vulnerable to A Man in the Middle attack, or MITM, is a situation wherein a malicious entity can read/write data that is being transmitted between two or more systems.

Likelihood

Medium

Risk

High
Recommandations

Define hadoop.rpc.protection to privacy in core-site.xml activates data encryption.
Data encryption on block data transfer

The Namennode gives the client the address of the first DataNode (DN) to read or write the block. The actual data transfer between the client and a DN is over Data Transfer Protocol.
Threat

  • Information leakage

Vulnerability

  • Unprotected network connections

  • Inadequate control of physical access

Asset

  • All HDFS components (servers and clients)

Impact

Block transfers between HDFS datanodes and clients are vulnerable to A Man in the Middle attack.

Likelihood

Medium

Risk

High
Recommandations

Change the actual settings:

  • dfs.encrypt.data.transfer → false

  • dfs.encrypt.data.transfer.algorithm → none

  • dfs.encrypt.data.transfer.cipher.suites → AES/CTR/NoPadding

And define:

  • dfs.encrypt.data.transfer to true in the hdfs-site.xml in order to activate data encryption for data transfer protocol of DataNode.

  • dfs.encrypt.data.transfer.algorithm to either 3des or rc4 to choose the specific encryption algorithm. If nothing is set, then the default on the system is used (usually 3DES.) While 3DES is more cryptographically secure, RC4 is substantially faster.

  • dfs.encrypt.data.transfer.cipher.suites to AES/CTR/NoPadding in order to activate AES encryption.
Data Encryption on HTTP

Data transfer between Web-console and clients are protected by using SSL(HTTPS).
Threat

  • Information leakage

Vulnerability

  • Unprotected network connections

  • Inadequate control of physical access

Asset

  • All HDFS components (servers and clients)

Impact

Block transfers between HDFS datanodes and clients are vulnerable to A Man in the Middle attack.

Likelihood

Medium

Risk

High
Recommandations

Change the actual settings:

  • dfs.http.policy → HTTP_ONLY

  • yarn.http.policy → HTTP_ONLY

  • mapreduce.jobhistory.http.policy → HTTP_ONLY

And define:

  • dfs.http.policy to either HTTPS_ONLY or HTTP_AND_HTTPS in hdfs-site.xml to enable SSL for web console of HDFS daemons.

  • yarn.http.policy to HTTPS_ONLY in yarn-site.xml to enable SSL for web console of YARN daemons.

  • mapreduce.jobhistory.http.policy to HTTPS_ONLY in mapred-site.xml to enable SSL for web console of MapReduce JobHistory server.

3.2. Authentication

3.2.1. Kerberos

Hadoop uses Kerberos as the basis for strong authentication and identity propagation for both user and services. Kerberos is a third party authentication mechanism, in which users and services rely on a third party - the Kerberos server - to authenticate each to the other.
Threat

  • Information leakage

Vulnerability

  • Unprotected network connections

  • Inadequate control of physical access

Asset

  • All HDFS components (servers and clients)

Impact

No user identity verification in the cluster

Likelihood

Medium

Risk

High
Recommandations

Change the actual settings:

  • hadoop.security.authentication=simple

  • hadoop.security.authorization=false

And define:

  • hadoop.security.authentication=kerberos

  • hadoop.security.authorization=true

3.2.2. SPNEGO

By default, access to the HTTP-based services and UIs for the cluster are not configured to require authentication. Kerberos authentication can be configured for the Web UIs for HDFS, YARN, MapReduce2, HBase, Oozie, Falcon, and Storm.
Threat

  • Information leakage

  • Theft

  • Vandalism

  • Industrial espionage

  • Malicious code

  • Access by unauthorized persons

  • Damage caused by a third party

Vulnerability

  • Lack of systems for identification and authentication

Asset

  • All components (servers and clients)

Impact

No user identity verification in the cluster

Likelihood

Medium

Risk

High
Recommandations

Change the actual settings:

  • hadoop.http.authentication.simple.anonymous.allowed → true

  • hadoop.http.authentication.signature.secret.file → none

  • hadoop.http.authentication.type → none

  • hadoop.http.authentication.kerberos.keytab → none

  • hadoop.http.authentication.kerberos.principal → none

  • hadoop.http.filter.initializers → none

  • hadoop.http.authentication.cookie.domain → none

And define in core-site.xml:

  • hadoop.http.authentication.simple.anonymous.allowed → false

  • hadoop.http.authentication.signature.secret.file → /etc/security/http_secret

  • hadoop.http.authentication.type → kerberos

  • hadoop.http.authentication.kerberos.keytab → /etc/security/keytabs/spnego.service.keytab

  • hadoop.http.authentication.kerberos.principal → HTTP/_HOST@EXAMPLE.COM

  • hadoop.http.filter.initializers → org.apache.hadoop.security.AuthenticationFilterInitializer

  • hadoop.http.authentication.cookie.domain → hortonworks.local

3.3. Authorization

Authorization defines access rights to resources, or privileges of use (permissions) for users. Service Level Authorization is the initial authorization mechanism to ensure clients connecting to a particular Hadoop service have the necessary, pre-configured, permissions and are authorized to access the given service.
Threat

  • Information leakage

Vulnerability

  • Inadequate control of access

Asset

  • All (Yarn, Hive, HDFS…​)

Impact

Service Level Authorization is performed much before to other access control checks such as file-permission checks, access control on job queues etc. By default, service-level authorization is disabled for Hadoop, which means anyone is allowed to submit jobs or see content in HDFS.

Likelihood

Low

Risk

High
Recommandations

Change the actual settings in core-site.xml:

  • hadoop.security.authorization=true

3.3.1. Ranger

Once a user has been authenticated, their access rights must be determined. Authorization defines user access rights to resources. For example, a user may be allowed to create a policy and view reports, but not allowed to edit users and groups. You can use Ranger to set up and manage access to Hadoop services. Ranger enables you to create services for specific Hadoop resources (HDFS, HBase, Hive, etc.) and add access policies to those services.
Threat

Information leakage

Vulnerability

Inadequate control of access

Asset

All (Yarn, Hive, HDFS…​)

Impact

Users are not restricted in terms or resource or access.

Likelihood

Low

Risk

High
Recommandations

Change the actual settings in yarn-site.xml:

  • yarn.acl.enable=true

  • yarn.authorization-provider= org.apache.ranger.authorization.yarn.authorizer.RangerYarnAuthorizer

Change the actual settings in hdfs-site.xml:

  • dfs.namenode.inode.attributes.provider.class= org.apache.ranger.authorization.hadoop.RangerHdfsAuthorizer

Change the actual settings in hive-site.xml:

  • hive.security.authorization.enabled=true

  • hive.security.authorization.manager= org.apache.ranger.authorization.hive.authorizer.RangerHiveAuthorizerFactory

Impact

Likelihood

  • Placing a firewall between cluster nodes will impact the cluster performance.

NIC Bonding (also known as NIC Teaming or Link Aggregation) provides a method for aggregating multiple network interfaces into a single logicalbonded interface. The behavior of the bonded interfaces depends upon the mode; generally speaking, modes provide either hot standby or load balancing services.
Threat

4.1.4. Multihoming

Multihoming is a strategy useful to assure network performance and predictability for both applications and management purposes. In multihomed networks the cluster nodes are connected to more than one network interface. There could be multiple reasons for doing so.

  • Security: Security requirements may dictate that intra-cluster traffic be confined to a different network than the network used to transfer data in and out of the cluster.

  • Performance: Intra-cluster traffic may use one or more high bandwidth interconnects like Fiber Channel, Infiniband or 10GbE.

  • Failover/Redundancy: The nodes may have multiple network adapters connected to a single network to handle network adapter failure.
Threat

Interruption of service

Vulnerability

Uncontrolled use of network

Asset

Servers

Impact

Network congestion, direct impact of bandwidth used by clients on admin operations like replication and cluster operations.

Likelihood

Low

Risk

Medium (private cluster, non shared, dedicated for one project)
Recommandations

Change the actual settings in core-site.xml to:

  • dfs.client.use.datanode.hostname=true

  • dfs.datanode.use.datanode.hostname=true

  • dfs.namenode.rpc-bind-host=0.0.0.0

  • dfs.namenode.servicerpc-bind-host=0.0.0.0

  • dfs.namenode.http-bind-host=0.0.0.0

  • dfs.namenode.https-bind-host=0.0.0.0

Change the actual settings in yarn-site.xml to:

  • yarn.resourcemanager.bind-host=0.0.0.0

  • yarn.nodemanager.bind-host=0.0.0.0

  • yarn.timeline-service.bind-host=0.0.0.0

Change the actual settings in mapred-site.xml to:

  • mapreduce.jobhistory.bind-host=0.0.0.0

4.2. High availbility

4.2.1. Yarn

The ResourceManager (RM) is responsible for tracking the resources in a cluster, and scheduling applications (e.g., MapReduce jobs). Prior to Hadoop 2.4, the ResourceManager is the single point of failure in a YARN cluster. The High Availability feature adds redundancy in the form of an Active/Standby ResourceManager pair to remove this otherwise single point of failure.
Threat

Interruption of service

Vulnerability

Lack of redundancy

Asset

Yarn ResourceManager

Impact

Single point of failure: in case of crash, all the Yarn resources provided by the NodeManagers will be unavailable.

Likelihood

High

Risk

High
Recommandations

Change the actual settings to:

  • yarn.resourcemanager.zk-address= sd00955105.gsb.loc:2181,sd00955111.gsb.loc:2181,sd00955112.gsb.loc:2181

  • yarn.resourcemanager.ha.enabled=true

  • yarn.resourcemanager.ha.rm-ids=rm1,rm2

  • yarn.resourcemanager.hostname.rm1=sd00955105.gsb.loc

  • yarn.resourcemanager.hostname.rm2=sd00955111.gsb.loc

  • yarn.client.failover-proxy-provider= org.apache.hadoop.yarn.client.RequestHedgingRMFailoverProxyProvider

4.2.2. HDFS

Prior to Hadoop 2.0.0, the NameNode was a single point of failure (SPOF) in an HDFS cluster. Each cluster had a single NameNode, and if that machine or process became unavailable, the cluster as a whole would be unavailable until the NameNode was either restarted or brought up on a separate machine. With a HA setup with quorum there are two NameNodes configured with one working as the active and the other as the standby instance.
Threat

Interruption of service

Vulnerability

Lack of redundancy

Asset

HDFS Namenode

Impact

Single point of failure: in case of crash, all the HDFS capacity provided by the DataNodes will be unavailable.

Likelihood

High

Risk

High
Recommandations

Change the actual settings to:

  • fs.defaultFS=hdfs://lab-ey

  • dfs.nameservices=lab-ey

  • dfs.ha.namenodes.lab-ey=nn201,nn202

  • dfs.namenode.rpc-address.lab-ey.nn01=nn01.hdp:8020

  • dfs.namenode.servicerpc-address.lab-ey.nn01=nn01.hdp:54321

  • dfs.namenode.http-address.lab-ey.nn01=nn01.hdp:50070

  • dfs.namenode.https-address.lab-ey.nn01=nn01.hdp:50470

  • dfs.namenode.rpc-address.lab-ey.nn02=nn02.hdp:8020

  • dfs.namenode.servicerpc-address.lab-ey.nn02=nn02.hdp:54321

4.2.3. Knox

Apache Knox Gateway is used to help ensure perimeter security and provides a central gateway for Hadoop REST APIs that have varying degrees of authorization, authentication, SSL, and SSO capabilities to enable a single access point for Hadoop.
Threat

Interruption of service

Vulnerability

Lack of redundancy

Asset

Knox Gateway server

Impact

Single point of failure: in case of crash, all clients using Knox will be unavailable.

Likelihood

High

Risk

High
Recommandations

  • Install Knox only on Edge servers.

  • Use multiple instances of Knox Gateway server.

  • Use a load-balancer (f5, HAProxy, Apache+mod_proxy…​) in front of all the instances with a Sticky Session mode.

  • Knox provides basic failover and retry functionality for REST API calls made to a service when service HA has been configured and enabled. To enable HA functionality in Knox the following configuration must be added to the topology file:

    maxFailoverAttempts=3;failoverSleep=1000;enabled=true;

Then, for each service configured, use a HA configuration by adding an additional URL:

<service>
  <role>{COMPONENT}</role>
  <url>http://{host1}:50070/{component}</url>
  <url>http://{host2}:50070/{component}</url>
</service>

The additional URLs for standby nodes should be added to the list. The active URL (at the time of configuration) should ideally be added at the top of the list.

4.4.1. Memory

Transparent Hugepages

Transparent Hugepages is a Linux kernel feature intended to improve performance by making more efficient use of your processor’s memory-mapping hardware. It is enabled ("enabled=always") by default in most Linux distributions.
Threat

Significant performance impact compromising platfrom stability

Vulnerability

Inadequate segregation of process using memory

Asset

Servers

Impact

Transparent Hugepages gives some applications a small performance improvement (~ 10% at best, 0-3% more typically), but can cause significant performance problems, or even apparent memory leaks at worst.

Likelihood

Low

Risk

Medium
Recommandations

  • Transparent Hugepages should be disabled for all nodes on the cluster.
